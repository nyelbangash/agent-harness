# Plan: Provenance marker on all harness-authored GitHub bodies (#1)

## Problem restatement

The harness posts to GitHub (issue comments, PR bodies) using the owner's PAT, so every harness-written body appears under the owner's GitHub login. Any future lane that reads comments and acts on "owner-authored" text would ingest its own prior output as a trusted human instruction — a trust-confusion vulnerability. The fix is a machine-readable provenance marker that riders append to every outgoing body and readers use to filter before acting. Two specific write paths exist today and must be covered: the plan-packet comment written by `PlanWorker.deliver/4` and the PR body built by `ImplementWorker.pr_body/4`.

## Implementation plan

### Step 1 — New module `harness/lib/harness/github/provenance.ex`

Create a pure, side-effect-free utility module with three public functions:

```
defmodule Harness.GitHub.Provenance do
  @moduledoc """
  Convention: every harness-authored GitHub body must pass through stamp/3;
  readers must filter with harness_authored?/1 before treating owner-authored text as instructions.
  """

  @marker_re ~r/<!-- harness:v1 kind=(?<kind>[^\s>]+) ref=(?<ref>[^\s>]+) -->/

  def stamp(body, kind, ref) do
    String.trim_trailing(body) <> "\n<!-- harness:v1 kind=#{kind} ref=#{ref} -->"
  end

  def harness_authored?(body), do: body =~ @marker_re

  def parse(body) do
    case Regex.named_captures(@marker_re, body) do
      %{"kind" => kind, "ref" => ref} -> {:ok, %{version: "v1", kind: kind, ref: ref}}
      nil -> :error
    end
  end
end
```

- The marker is an HTML comment so it renders to nothing in GitHub Markdown.
- `kind` and `ref` values are unquoted tokens (`[^\s>]+`); both workers pass simple strings that contain no whitespace or `>`.
- The version literal `"v1"` is hard-coded in `parse/1`'s return; it does not need to be in the marker regex because the regex already anchors on the `v1` prefix literal in the pattern string — future versions bump the literal in `stamp/3` and add a new regex.
- No external dependencies.

### Step 4 — New test file `harness/test/harness/github/provenance_test.exs`

```elixir
defmodule Harness.GitHub.ProvenanceTest do
  use ExUnit.Case, async: true

  alias Harness.GitHub.Provenance

  test "stamp → harness_authored? → parse round-trip" do
    stamped = Provenance.stamp("Hello!", "plan", "run-42")
    assert Provenance.harness_authored?(stamped)
    assert {:ok, %{version: "v1", kind: "plan", ref: "run-42"}} = Provenance.parse(stamped)
  end

  test "unmarked body returns false / :error" do
    refute Provenance.harness_authored?("Just a regular comment.")
    assert :error = Provenance.parse("Just a regular comment.")
  end

  test "marker is a valid HTML comment that renders to nothing" do
    stamped = Provenance.stamp("body", "pr", "run-99")
    marker = stamped |> String.split("\n") |> Enum.find(&String.starts_with?(&1, "<!--"))
    assert marker =~ ~r/^<!-- harness:v1 kind=\S+ ref=\S+ -->$/
  end

  test "original body content is preserved" do
    body = "## Summary\n\nSome text."
    stamped = Provenance.stamp(body, "plan", "run-1")
    assert String.starts_with?(stamped, "## Summary")
    assert String.contains?(stamped, "Some text.")
  end
end
```

## Alternatives considered

**Attribute-level stamp (metadata field instead of body suffix):** GitHub's API does not support custom metadata on comments or PRs. All machine-readable information must live in the body itself.

**Separate machine account (bot login):** The issue explicitly lists this as a non-goal. It would also require a second PAT, a new GitHub account, and significant infra work — vastly disproportionate to the problem.

**Check author login at read time:** Reading the GitHub API `user.login` on each comment before acting is an N+1 API call per comment (or requires storing the harness's login), and still fails when the owner manually writes a comment that exactly mimics harness formatting. The in-body marker is cheaper, zero-API-call, and forgery-resistant by design (an owner would have to deliberately include the magic string).

**Different marker format (e.g., YAML front-matter, JSON block):** A standard HTML comment (`<!-- ... -->`) is the only format GitHub renders as invisible. YAML front-matter is GitHub-Pages-specific and would appear as raw text in issue comments. A fenced code block would be visible.
